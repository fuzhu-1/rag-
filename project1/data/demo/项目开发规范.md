# 软件项目开发规范 V3.2

## 一、代码规范

### 1.1 命名规范
- **变量/函数**：使用蛇形命名法 `snake_case`
- **类名**：使用帕斯卡命名法 `PascalCase`
- **常量**：使用全大写蛇形 `UPPER_SNAKE_CASE`
- **私有成员**：以下划线前缀 `_private_method`

### 1.2 代码格式
项目使用 `ruff` 作为代码检查工具，配置如下：
- 行宽限制：100 字符
- 缩进：4 个空格（禁止使用 Tab）
- 导入排序：使用 `isort`（标准库 → 第三方 → 本地）

### 1.3 类型注解
所有公共函数必须包含完整的类型注解：
```python
def process_data(data: list[dict[str, Any]], threshold: float = 0.8) -> pd.DataFrame:
    ...
```

### 1.4 文档字符串
使用 Google 风格的 docstring，必须包含参数说明、返回值说明和异常说明。

## 二、Git 工作流

### 2.1 分支策略
- `main`：生产环境分支，只接受来自 `release` 和 `hotfix` 的 PR
- `develop`：开发主分支
- `feature/<name>`：功能分支
- `bugfix/<name>`：修复分支
- `release/<version>`：发布分支
- `hotfix/<name>`：紧急修复分支

### 2.2 Commit 规范
使用语义化提交信息：`<type>(<scope>): <description>`

类型包括：
- `feat`：新功能
- `fix`：修复 Bug
- `docs`：文档更新
- `style`：代码格式
- `refactor`：代码重构
- `test`：测试相关
- `chore`：构建/工具变更

### 2.3 Code Review 要求
- 每个 PR 至少需要 2 人 Review 通过
- 关键模块需要技术负责人审批
- PR 合并前所有 CI 检查必须通过

## 三、测试规范

### 3.1 测试覆盖率
- 核心业务逻辑：≥ 85% 行覆盖率
- 工具函数：≥ 90% 行覆盖率
- UI 组件：≥ 70% 行覆盖率

### 3.2 测试分类
```python
# 单元测试：文件名 test_*.py
# 集成测试：文件名 integration_test_*.py
# 端到端测试：文件名 e2e_test_*.py
```

### 3.3 Mock 策略
外部依赖（数据库、API、文件系统）必须 Mock，内部纯函数禁止 Mock。

## 四、API 设计规范

### 4.1 RESTful 规范
- 使用复数名词：`/api/users`, `/api/orders`
- HTTP 方法语义：
  - `GET`：查询
  - `POST`：创建
  - `PUT`：全量更新
  - `PATCH`：部分更新
  - `DELETE`：删除

### 4.2 响应格式
```json
{
    "code": 0,
    "message": "success",
    "data": {},
    "request_id": "uuid"
}
```

### 4.3 错误码规范
| 错误码 | 含义 | HTTP 状态码 |
|--------|------|-------------|
| 0 | 成功 | 200 |
| 1001 | 参数错误 | 400 |
| 1002 | 认证失败 | 401 |
| 1003 | 权限不足 | 403 |
| 1004 | 资源不存在 | 404 |
| 2001 | 服务器内部错误 | 500 |
| 2002 | 服务不可用 | 503 |

## 五、数据库规范

### 5.1 命名规范
- 表名：小写蛇形，复数形式（`users`, `order_items`）
- 字段名：小写蛇形（`created_at`, `updated_by`）
- 索引名：`idx_<table>_<column>`
- 唯一索引：`udx_<table>_<column>`

### 5.2 必需字段
每个业务表必须包含以下字段：
```sql
id          BIGINT PRIMARY KEY AUTO_INCREMENT,
created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
updated_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
is_deleted  TINYINT NOT NULL DEFAULT 0
```

### 5.3 索引规范
- 查询频繁的字段建立索引
- 联合索引列数不超过 3 个
- 避免在低基数列上建索引（如性别、布尔值）

## 六、部署与运维

### 6.1 环境配置
- 开发环境：Docker Compose 本地部署
- 测试环境：Kubernetes（命名空间 `test`）
- 生产环境：Kubernetes（命名空间 `prod`），至少 3 个副本

### 6.2 日志规范
- 使用结构化日志（JSON 格式）
- 日志级别使用规范：
  - `DEBUG`：开发调试
  - `INFO`：关键业务流程节点
  - `WARNING`：可恢复异常
  - `ERROR`：需人工介入的错误

### 6.3 监控告警
- API 响应时间 > 2 秒：告警
- 5xx 错误率 > 1%：告警
- 内存使用率 > 85%：告警
- 磁盘使用率 > 90%：告警
- 数据库连接池耗尽：紧急告警

### 6.4 CI/CD 流水线
1. 代码检查（ruff, mypy）
2. 单元测试 + 覆盖率
3. 安全漏洞扫描
4. 构建 Docker 镜像
5. 部署到测试环境
6. 冒烟测试
7. 人工审批（生产环境）
8. 滚动更新部署
